# Security Policy

P2poolBTC is experimental. Fork coins have no promised value, but the project
interacts with two live-value trust boundaries:

- Experiment 1 accepts every transaction and script class supported by its
  pinned Bitcoin Core engine, including spends of inherited mainnet UTXOs.
  Address encodings and private keys are shared with Bitcoin mainnet. Mixed-input
  replay protection prevents ordinary exact transaction replay; it does not
  protect a disclosed or reused key. Never import a valuable mainnet key.
- Idena is a live chain, not a disposable testnet. Identity signatures are
  public, and delegation, stake, validation, transactions, and contract calls
  can affect real IDNA or identity state. P2poolBTC needs only a public address
  and a domain-separated signature; never provide an Idena key, backup,
  password, or API key.

## Private Reporting

Do not open a public issue for a suspected vulnerability or privacy leak. Open
the repository's [Security page](https://github.com/ubiubi18/P2poolBTC/security)
and select **Report a vulnerability** when private reporting is available. If
that option is unavailable, contact the repository owner through their GitHub
profile without sending technical evidence, and arrange a private channel
before sharing reproduction details.

Report the minimum information required to reproduce the problem:

- affected commit and component;
- impact and required attacker access;
- sanitized reproduction steps;
- whether fork or mining services were stopped;
- a minimal proof that contains no real keys, wallets, identities, addresses,
  API credentials, cookies, peer endpoints, or personal information.

Do not upload raw logs, `.env` files, databases, core dumps, identity keystores,
wallet files, report bundles, or screenshots until every field has been
reviewed. If a secret may have been exposed, revoke or rotate it before further
testing.

## Immediate Stop Conditions

Stop the affected service and preserve its datadir if you observe an activation
mismatch, replay-root divergence, invalid signature acceptance, unauthorized
RPC access, unexpected Bitcoin mainnet submission, or unintended disclosure of
participant data. Also stop before signing if a fork transaction does not
contain the expected fork-created replay-separator input or if any UI confuses
an inherited explorer balance with a fork-wallet balance.

There is currently no production security guarantee or real-value bug bounty.

## Host Firewall Baseline

Debian hosts should use the dedicated nftables policy generated by
`scripts/pohw-host-firewall.py`. The default operation is a read-only audit; it
does not install packages or alter the host. The policy owns only the
`inet pohw_host_firewall` table and never flushes unrelated firewall tables.

Audit the current table and TCP listeners:

```sh
sudo python3 scripts/pohw-host-firewall.py \
  --ssh-port 22 \
  --public-p2p-port 8333 \
  --public-p2p-port 40405 \
  --public-p2p-port 40406 \
  --public-p2p-port 40409 \
  --status
```

Use the same intended SSH and public-P2P allowlist for every later audit. A
different list is reported as managed-rule drift rather than silently accepted.

Applying requires an explicit acknowledgement. List every SSH port that must
remain reachable on a public interface and every intentionally public P2P TCP
port. No public SSH or P2P port is inferred from a wildcard listener. Omit
`--ssh-port` when SSH is reachable only through Tailscale:

```sh
sudo python3 scripts/pohw-host-firewall.py \
  --ssh-port 22 \
  --public-p2p-port 8333 \
  --public-p2p-port 40405 \
  --public-p2p-port 40406 \
  --public-p2p-port 40409 \
  --apply I_UNDERSTAND_THIS_REPLACES_THE_POHW_FIREWALL_TABLE
```

Adjust that allowlist to the services actually enabled on the host. The tool
runs `nft --check` before one atomic ruleset load. It preserves established and
related connections, loopback, ICMP/ICMPv6, `tailscale0`, and the selected SSH
ports. Every other inbound public-interface flow is dropped. Bitcoin and Idena
RPC, dashboard UI/API, Stratum, explorer backends, and fork control APIs must
remain loopback-only or be reached through SSH/Tailscale.

The listener audit reports wildcard sensitive services even when the firewall
currently blocks them. It also reports sensitive services bound to a specific
non-loopback address; only a loopback bind passes that defense-in-depth check.
Fix the service bind address. An existing sensitive connection is not terminated because established
connections are deliberately preserved. Confirm an out-of-band recovery path
and the correct SSH port before applying a firewall remotely. The reference
policy is `deploy/nftables/pohw-host-firewall.nft.example`.

`--apply` changes only the running kernel ruleset. It intentionally does not
rewrite `/etc/nftables.conf`, enable a boot service, or claim reboot
persistence because those actions could overwrite unrelated host policy. The
JSON report therefore exposes `persistenceManaged: false`.

For explicit reboot persistence, install the dedicated helper, strict JSON
policy, and systemd unit. Review and edit the port arrays before enabling it;
the example uses Tailscale-only SSH and opens only the listed P2P services:

```sh
sudo install -D -o root -g root -m 0755 \
  scripts/pohw-host-firewall.py /usr/local/libexec/pohw-host-firewall.py
sudo install -D -o root -g root -m 0600 \
  deploy/nftables/pohw-host-firewall.json.example \
  /etc/pohw/host-firewall.json
sudo install -D -o root -g root -m 0644 \
  deploy/systemd/pohw-host-firewall.service \
  /etc/systemd/system/pohw-host-firewall.service
sudoedit /etc/pohw/host-firewall.json
sudo systemctl daemon-reload
sudo systemctl enable --now pohw-host-firewall.service
sudo systemctl reload pohw-host-firewall.service
sudo /usr/bin/python3 -I /usr/local/libexec/pohw-host-firewall.py \
  --config /etc/pohw/host-firewall.json --status
```

The boot service applies only `inet pohw_host_firewall`; it does not flush or
replace unrelated nftables tables. Keep an out-of-band recovery path and audit
again after every reboot before treating the host as externally reachable.

When invoked through `sudo`, the helper accepts only the root-owned Debian
executables `/usr/sbin/nft` and `/usr/bin/ss`; command-line path overrides are
available only to unprivileged tests and audits. This prevents a privileged
caller from turning `--nft-bin` or `--ss-bin` into arbitrary root code
execution.
